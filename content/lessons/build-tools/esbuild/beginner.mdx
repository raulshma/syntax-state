## Why esbuild is So Fast

esbuild is 10-100x faster than traditional bundlers. Here's why:

### 1. Written in Go, Not JavaScript

```
JavaScript Bundler:
üê¢ Single-threaded ‚Üí Process files one by one ‚Üí Slow

esbuild (Go):
üöÄ Multi-threaded ‚Üí Process files in parallel ‚Üí FAST!
```

<KeyConcept title="The Language Advantage">
JavaScript runs in a single thread and has garbage collection pauses. Go compiles to native machine code, uses all CPU cores simultaneously, and has minimal runtime overhead. This architectural difference makes esbuild fundamentally faster.
</KeyConcept>

### 2. Everything in One Pass

```
Traditional Bundler:
Pass 1: Parse ‚Üí Pass 2: Transform ‚Üí Pass 3: Bundle ‚Üí Pass 4: Minify
(Each pass reads and writes the entire codebase)

esbuild:
Parse + Transform + Bundle + Minify (ALL AT ONCE!)
(Single pass through the code)
```

## Getting Started with esbuild

### Installation

```bash
# Install esbuild
npm install esbuild

# Or globally
npm install -g esbuild
```

### Your First Build

```bash
# Bundle a file (CLI)
esbuild src/index.js --bundle --outfile=dist/bundle.js

# With minification
esbuild src/index.js --bundle --minify --outfile=dist/bundle.js

# Watch mode
esbuild src/index.js --bundle --watch --outfile=dist/bundle.js
```

### Using the JavaScript API

```javascript
// build.js
import * as esbuild from 'esbuild'

await esbuild.build({
  entryPoints: ['src/index.js'],
  bundle: true,
  outfile: 'dist/bundle.js',
})

console.log('Build complete! ‚ö°')
```

Run it:
```bash
node build.js
# Build complete! ‚ö° (in milliseconds!)
```

## Basic Configuration

```javascript
// build.js
import * as esbuild from 'esbuild'

await esbuild.build({
  // Entry points (can be multiple)
  entryPoints: ['src/index.js', 'src/worker.js'],
  
  // Bundle all imports into output
  bundle: true,
  
  // Output location
  outdir: 'dist',
  
  // Minify for production
  minify: true,
  
  // Generate source maps
  sourcemap: true,
  
  // Target environment
  target: ['es2020', 'chrome80', 'firefox78'],
  
  // Output format: 'iife', 'cjs', or 'esm'
  format: 'esm',
})
```

## Handling Different File Types

esbuild uses "loaders" to handle different file types:

```javascript
await esbuild.build({
  entryPoints: ['src/index.tsx'],
  bundle: true,
  outfile: 'dist/bundle.js',
  
  loader: {
    '.png': 'file',      // Copy to output, return URL
    '.svg': 'dataurl',   // Inline as base64 data URL
    '.json': 'json',     // Parse as JSON
    '.css': 'css',       // Bundle CSS
    '.txt': 'text',      // Import as string
  },
})
```

### Built-in Loaders

| Loader | Description |
|--------|-------------|
| `js` | JavaScript (default for .js) |
| `jsx` | JavaScript with JSX |
| `ts` | TypeScript |
| `tsx` | TypeScript with JSX |
| `css` | CSS files |
| `json` | JSON files |
| `text` | Plain text (as string) |
| `file` | Copy file, return URL |
| `dataurl` | Inline as base64 |

## When to Use esbuild

###  Great For:
- **Library bundling** - Fast builds, simple output
- **Development tooling** - Powers Vite's dev server
- **Simple applications** - When you don't need complex plugins
- **TypeScript compilation** - Fastest TS ‚Üí JS conversion

### Ô∏è Consider Alternatives When:
- You need complex code transformations
- You require extensive plugin ecosystem
- You need CSS Modules with scoped names
- You need advanced code splitting strategies

<InfoBox type="tip" title="esbuild + Other Tools">
esbuild is often used as a component in other tools! Vite uses esbuild for dependency pre-bundling and TypeScript compilation. You get esbuild's speed without giving up other features.
</InfoBox>
