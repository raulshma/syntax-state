# Working with remotes (Intermediate)

<InfoBox type="tip">
Intermediate remotes = understanding **upstream tracking** and avoiding accidental merges.
</InfoBox>

## Inspect and manage remotes

List remotes:

```bash
git remote -v
```

Add a remote:

```bash
git remote add origin https://github.com/OWNER/REPO.git
```

Show remote details:

```bash
git remote show origin
```

## Tracking branches and upstream

When your branch tracks a remote branch, you can use simple commands like `git pull` and `git push` without extra args.

Set upstream the first time you push:

```bash
git push -u origin main
```

Then later:

```bash
git pull
```

```bash
git push
```

## Prefer predictable pull behavior

Many developers pick one of these patterns:

### Fast-forward only (safe default)

```bash
git config --global pull.ff only
```

### Rebase on pull (keeps linear history)

```bash
git config --global pull.rebase true
```

## Forks and upstream remotes (common in open source)

- `origin`: your fork
- `upstream`: the original project

```bash
git remote add upstream https://github.com/ORIGINAL/REPO.git
```

Fetch upstream:

```bash
git fetch upstream
```

Bring changes into your branch (example):

```bash
git switch main

git merge upstream/main
```

## Summary

<KeyConcept title="Intermediate remote workflow">
- Use upstream tracking (`git push -u`) for simpler pulls/pushes
- Choose a pull strategy (ff-only or rebase) intentionally
- Use `upstream` remote when working with forks
</KeyConcept>
